# Crystal-LoRA v4 — what to build next

## Where v3 landed

Held-out eval (30 natural-language Crystal tasks, idiom + `crystal build --no-codegen` gate):

| Model | Idiom | Compile pass | Total |
|---|---|---|---|
| **crystal-qwen-v3** | **+76** | **26/28 (93 %)** | **+206** |
| jaimef/crystal-qwen3.6-30b (v2) | +67 | 21/28 (75 %) | +172 |
| qwen3-coder:30b (base) | +71 | 21/28 (75 %) | +176 |

Similarity eval (74 DPO pairs, lean toward "chosen" idiomatic Crystal):

| Model | Token lean | Char lean | Pairs leaning chosen |
|---|---|---|---|
| **v3** | **+6.547** | +3.013 | 66/74 |
| base | +6.077 | **+3.107** | 74/74 |
| v2 | +2.345 | −0.906 | 52/74 |

**Key signal**: v2 was a regression — lost to base on idiom (+67 vs +71) and had **negative** char-lean
(−0.906 means it character-matched the *rejected* Ruby-style answer better than the chosen one).
v3 is the first checkpoint in this lineage that beats vanilla Qwen3-Coder at Crystal.

**The harder truth**: v3-vs-v2 = +34 total looks great because v2 was broken. v3-vs-base = +30
total comes mostly from compile-gate (+5 pass) and only +5 idiom. Without a new training signal,
v4 risks being v3+1 — measurable but unexciting.

## Highest-leverage v4 ideas

### 1. 10× the DPO corpus  (~ 1 day, no GPU)

DPO is the smallest stage (74 pairs) and visibly shaped v3's idiom edge. Mechanically generate
rejected variants by injecting common Crystal-mistake patterns into known-good Crystal:

| Pattern (correct) | Inject as rejected |
|---|---|
| `property name : String` | `attr_accessor :name` |
| `Hash(String, Int32)` | `Hash<String, Int32>` |
| `String?` / `String \| Nil` | `Optional<String>` / `Maybe(String)` |
| `spawn { ... }` + `Channel(T)` | `Thread.new { ... }` + `Mutex.new` |
| `require "spec"` / `describe`/`it`/`.should` | `require "rspec"` / `RSpec.describe` / `expect(...).to eq(...)` |
| `responds_to?(:foo)` | `respond_to?(:foo)` |
| `&.` safe-nav | `&block` / `dig(...)` / `block_given?` |
| `def initialize(@name : String)` | `def initialize(name); @name = name; end` |
| `record Money, amount : Int64, currency : String` | `class Money; attr_*; end` / `Struct.new` |
| typed `def f(x : Int32) : Int32` | untyped `def f(x)` |

Each chosen example yields 3–5 rejected variants. Target 500–1000 pairs (vs v3's 74). Cheap,
parallelizable, *directly* targets the failure modes the eval is measuring.

**Risk**: synthetic rejecteds may be too easy to discriminate, leaving the harder real-world
mistakes untrained. Mitigate by mixing with a smaller pool (~100) of LLM-mined rejecteds drawn
from actual GitHub Crystal-from-Ruby porting commits.

### 2. Compile-error SFT — feed the compiler  (~ 3 days, needs Crystal in pipeline)

Mine ~1000 examples of the form:

```
{snippet that fails to compile}
↓
{full crystal build error stderr}
↓
{corrected snippet that compiles}
```

Sources:
- Synthetically corrupt known-good Crystal (drop `: Type`, swap `String?` → `String`, miss
  `require`, use Ruby-style `<<` on hash, etc.) — generates compile errors with known fixes.
- Mine from GitHub Crystal commits where the message contains "fix", "compile", "type
  error" — extract the diff as the snippet→fix pair, run pre-commit through the compiler to
  capture the error.

v3 still misses 2/28 compile gates and lost char-similarity to base. Teaching the model to
*fix its own type errors* attacks both axes. This is the only candidate that gives the model
a *new* training signal (compile-gate as data, not just as eval).

**Risk**: high engineering cost (Crystal compiler in mining loop, error stderr is noisy and
needs cleanup). Slow to iterate.

## Lower-leverage / probably skip

- **Full fine-tune instead of LoRA** — overtraining risk on a 30B model at our data size,
  marginal gain.
- **Higher LoRA rank** — already r=64. Diminishing returns past that for our token count.
- **Base model swap** (Qwen3.5-Coder if/when it ships) — that's "v4 of Crystal-Lora" but
  really just a base bump. Retraining everything for someone else's release.

## The one thing to do *regardless* of v4

Publish a **Q4_K_M v3** alongside the Q8_0. The current v3-vs-base 30-point margin includes
a quantization confound (v3 Q8_0 vs base Q4_K_M). A Q4_K_M v3 would either:
- **Confirm** v3 wins on equal footing → strong signal for the training pipeline.
- **Expose** the compile-gate edge as quant-noise → important to know before promising users
  v3 is meaningfully better.

Cost: ~10 minutes (`llama-quantize crystal-qwen-v3.gguf crystal-qwen-v3.Q4_K_M.gguf Q4_K_M`)
plus an eval re-run.

## Suggested order

1. **First**: Publish Q4_K_M v3 + re-run held-out eval. Confirms the v3 win is real, not
   quant-driven. (1 hour total.)
2. **Then**: DPO×10 — if v3 holds at Q4_K_M, the easy 10× DPO is the obvious next layer.
   Train DPO-only on top of the existing v3 merged checkpoint (no need to redo CPT/SFT).
   ~1 H200-hour to re-run just the DPO stage on the v3 base.
3. **Only if (2) shows real gains**: compile-error SFT. The engineering investment is high
   enough that it should ride a confirmed-good DPO foundation rather than be parallel
   speculation.

## What we'd need to know is "real progress"

Set the bar before spending compute:

- Held-out idiom: **v4 idiom > v3 idiom + 10**. (v3 was +9 over v2 — anything less is noise.)
- Held-out compile: **28/28**. Lock it down. Right now 2 misses are eval-fragment artifacts;
  fixing the questions is also a viable win.
- Similarity char lean: **v4 char_lean > base char_lean** (i.e. > +3.107). v3 currently
  loses this to base.
- New eval: add ~30 *long-form* Crystal tasks (multi-file, shard-scaffolding, web framework
  setup) — current eval is too one-shot. Without long-form measurement, v4 changes won't be
  visible above noise.
